# Weekly Progress Tracker

## Purpose
Track planned versus actual preparation and force a clear pass/fail call every week.

## Study Steps
- Use this file during the Sunday review block.
- Fill concrete numbers first: solved count, mocks, defects, shipped project work.
- Mark pass or fail using the criteria in the file, then pick one recovery action.

## Weekly Score
| Area | Points | Scoring rule |
|---|---:|---|
| DSA | 25 | Target count met, 70 percent first-pass correctness, misses revised |
| Java/backend | 20 | One topic explained with internals and production failure modes |
| System design | 20 | One full design with trade-offs, scale, failures, observability |
| Project | 15 | Visible artifact shipped or benchmarked |
| Mock/communication | 10 | One mock or timed set completed with feedback |
| Behavioral | 10 | One story improved with metrics and reflection |

Pass: 75 or more with no zero in mock, DSA, or design.
Fail: below 75, skipped mock, or no mistake review.

## Interview Questions
- What did you plan, what did you actually finish, and what slipped?
- Which miss would fail an interview if repeated tomorrow?
- What evidence proves project progress this week?

## Common Mistakes
- Writing hours instead of outcomes.
- Marking a problem solved after reading the editorial.
- Not separating first-pass success from revision success.

## Self-Check
- [ ] I have exact problem counts and design prompts for this week.
- [ ] Every failed item has a next review date.
- [ ] The next week has one fewer vague task than this week.

## Practical Example
Example: W07 passes with 14 DSA solved, 3 revised misses, 1 tree mock, thread-pool notes, and an async project flow merged. It fails if the mock was skipped.
