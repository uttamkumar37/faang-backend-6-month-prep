# Interview Tracking System

## Purpose
Create a measurable feedback loop for SDE-II backend preparation across DSA, system design, Java depth, projects, mocks, and behavioral readiness.

## Study Steps
- Start every week in weekly-progress.md.
- Log every failed or slow problem in mistake-log.md within 24 hours.
- Update all scorecards monthly before opening the next monthly gate.
- Keep mock feedback in one place so repeated issues are visible.

## Files
| File | Use |
|---|---|
| weekly-progress.md | Weekly plan, actuals, and pass/fail review |
| mistake-log.md | DSA, design, backend, and communication defects |
| dsa-pattern-scorecard.md | Pattern-level coding strength |
| system-design-scorecard.md | Design-round scoring rubric |
| java-backend-scorecard.md | Java, Spring, JVM, concurrency, DB, Kafka, Redis depth |
| project-progress-tracker.md | Portfolio execution and interview story readiness |
| mock-feedback.md | Mock interview notes and action items |
| monthly-review-template.md | Month-end review before using evaluation gates |

## Weekly Usage Instructions
1. Monday: select 3 DSA patterns, 1 backend topic, 1 design prompt, and 1 story to practice.
2. Wednesday: check if the week is drifting; reduce scope instead of carrying vague debt.
3. Sunday: score outcomes, update the mistake log, and schedule exact repairs.

## Pass/Fail Criteria
- Weekly pass: 80 percent or more of planned study blocks completed, 1 mock or timed set done, and all high-severity mistakes logged.
- Monthly pass: gate score is 75 or higher and no critical category is below 60 percent.
- Final pass: SDE-II readiness bar is interview-ready or higher in every required area.

## Interview Questions
- How do you know whether your preparation is improving week over week?
- Which scorecard would expose weak Java backend depth fastest?
- What is your current highest-risk interview round?

## Common Mistakes
- Counting solved problems without revisiting misses.
- Skipping mock feedback because it feels subjective.
- Treating projects as progress even when they cannot be explained in metrics.

## Self-Check
- [ ] I can name my weakest DSA pattern this week.
- [ ] I have a written action item from the last mock.
- [ ] I can explain whether I passed or failed the week with numbers.

## Practical Example
Example: If you solved 12 problems but 5 failed due to boundary conditions, the week is not a coding pass until those 5 are re-solved without notes.
