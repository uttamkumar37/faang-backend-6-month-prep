# Monthly Review Template

## Purpose
Perform a hard month-end review before moving to the next phase of the six-month plan.

## Study Steps
- Use this file during the Sunday review block.
- Fill concrete numbers first: solved count, mocks, defects, shipped project work.
- Mark pass or fail using the criteria in the file, then pick one recovery action.

## Monthly Summary
| Month | DSA solved | DSA revised | Design mocks | Coding mocks | LLD drills | Java topics | Behavioral stories | Gate score |
|---|---:|---:|---:|---:|---:|---|---:|---:|
| | | | | | | | | |

## Checklist
- DSA target met with revision history.
- At least one design answer recorded or written end to end.
- Java/backend topic explained with internals and production failure modes.
- Project produced a visible artifact.
- Behavioral story bank improved with metrics.
- Top 3 mistakes have repair drills scheduled.

## Decision
- Pass: score 75+, no critical category below 60, next month can start.
- Conditional pass: score 65-74, start next month but reserve first 3 days for repair.
- Fail: below 65, spend one recovery week before advancing.

## Interview Questions
- What did this month prove about your readiness?
- Which category would cause a downlevel?
- What will you deliberately stop doing next month?

## Common Mistakes
- Advancing months based only on calendar date.
- Not preserving written evidence of design and project work.
- Ignoring low behavioral readiness until applications start.

## Self-Check
- [ ] I know my exact monthly score.
- [ ] I have a recovery plan for every area below 70.
- [ ] I can explain the next month focus in one paragraph.

## Practical Example
Example: Month 3 with strong DSA but production debugging score 45 should pause for JVM, pool, Kafka lag, Redis, and DB latency drills before adding advanced AI scope.
