# SDE-II Readiness Bar

## Purpose
Define what 10/10 readiness means for Java backend SDE-II and Senior Backend interviews.

## Study Steps
- Score each area from beginner to senior.
- Any area below interview-ready becomes a weekly repair target.
- Use this file during month 5 and month 6 reviews.
- Do not apply aggressively until all core backend areas are interview-ready or senior.

| Level | Bar |
|---|---|
| Beginner | Recognizes basic patterns after hints |
| Intermediate | Solves common easy/medium with bugs |
| Interview-ready | Solves mediums in 25-30 minutes with tests |
| Senior/SDE-II | Explains invariants, variants, trade-offs, and handles follow-ups |

Self-score checklist:
- [ ] I can explain this area without notes.
- [ ] I have solved or built at least one practical example.
- [ ] I can handle follow-up questions and trade-offs.
- [ ] I have mock or project evidence for this area.

## Final 10/10 Criteria
- No core area below interview-ready.
- DSA, system design, Java backend, production debugging, and behavioral leadership at senior/SDE-II level.
- Two projects can be defended with architecture, metrics, failures, and trade-offs.
- Mock interviews are consistently pass or strong pass.

## Interview Questions
- Which area would cause a downlevel today?
- What evidence proves senior-level production maturity?
- Can you connect project ownership to system design and debugging?

## Common Mistakes
- Self-scoring based on comfort instead of evidence.
- Ignoring production debugging and behavioral depth.
- Calling projects ready when they lack metrics or trade-offs.

## Self-Check
- [ ] Every area has a current level marked.
- [ ] Every weak area has a repair task.
- [ ] I can explain why I am ready for SDE-II in under 2 minutes.

## Practical Example
Example: Senior Kafka readiness is not 'I used Kafka'. It is explaining partitioning, ordering, consumer lag, retry/DLQ, idempotency, replay, and metrics in a real order-processing flow.
